The Mission

What We're Building

clinicIQ is an Agentic-as-a-Service platform for health experts. It automates the entire growth engine of a health expert's business — brand identity, strategy, content, funnels, emails, ads, webinars, programs, website, and podcast — using AI agents that know the expert deeply and produce finished, publish-ready work on demand.

The expert connects once. They talk to the platform in plain language. The platform builds everything. No prompts. No templates to fill out. No guessing what to post or write or build. Just outcomes.

The system knows the expert's voice, their avatar, their offer, their mechanism, and their brand. Every output — whether it's an Instagram caption, a webinar script, a Phase 2 supplement protocol, or a podcast runsheet — sounds like them, speaks to their specific audience, and fits precisely into their growth strategy.

"The expert never writes a caption, builds a funnel page, or touches a prompt. They just talk to the platform and it builds."
🎙️
Voice-Locked Output
Every word the platform writes runs through the expert's Voice DNA — extracted during onboarding and refined continuously. Outputs sound like the expert wrote them.
🔁
Self-Improving System
Layer 4 connects every data source — Meta Ads, GHL, email, social, podcast, program. The platform diagnoses underperformance, executes fixes, and gets measurably better every cycle.
💊
Prescription-Level Specificity
Program protocols aren't vague health advice. They're specific supplement names, doses, timing, and clinical rationale. If it's vague, it's not done.
📡
Built to Scale
From a solo practitioner launching their first program to Dr. Josh Axe scaling a podcast to 2M plays/month — same platform, same agents, depth activates automatically with growth.
🔒
White-Labeled Throughout
Patients never see clinicIQ. The course player, AI assistant, and all tools live at the expert's domain. clinicIQ is the engine, invisible under the hood.
🧠
One Brain, Many Hands
IQ Claw is the expert's persistent AI operator — it knows everything about the business and routes every request to the right specialist agent instantly.

Needs Resolution

8 Engineering Decisions Before Build

These are the open questions the engineering team needs to resolve. The spec is complete — these decisions determine how it gets built, not what gets built.

Decision 01
Agent Framework
LangGraph, CrewAI, OpenAI Agents SDK, or custom? All are production-viable in 2026. The choice is team capability and cost — not architecture quality.
LangGraph for complex orchestration graphs; CrewAI for simpler role-based teams
Decision 02
Model Tier Map
Which LLM/SLM handles which agent type? This is the most important cost-at-scale decision. Not all agents need GPT-4o. Many should use smaller, faster, cheaper models.
Voice-critical: Claude 3.5+ / GPT-4o. Extraction: smaller models. Classification: SLMs. Never premium for formatting.
Decision 03
Memory & State Backend
The Expert Dossier and IQ Profile are living documents accessed across sessions by dozens of agents. Where does this live? How is it accessed per-session? Most common production failure point in multi-agent systems.
Vector DB (semantic) + PostgreSQL (structured) + Redis (session state)
Decision 04
IQ Drive Tech Stack
Asset storage with rich metadata, semantic search, version history, and performance data overlay. Needs to be fast, scalable, and queryable by agents and Layer 4 simultaneously.
S3-compatible object storage + vector DB + PostgreSQL metadata + React/Next.js UI
Decision 05
IQ Course Player Stack
The Program Builder's native course player needs video hosting, completion tracking, a patient AI knowledge base, and white-label hosting at the expert's domain. Already recommended in spec.
React/Next.js + Mux + Puppeteer + PostgreSQL + vector DB — confirmed Step 16b
Decision 06
Agent Deployment Model
Some agents are always-on (IQ Claw, Data Ingestion). Some are episodic/triggered (Builder agents). Mixed deployment model is appropriate — but needs to be defined up front.
Serverless for triggered agents; containers for always-on agents
Decision 07
IQ Claw Interface Layer
IQ Claw is a persistent panel accessible from every screen. What's the technical implementation? Web app overlay? Native widget? This affects the entire UX architecture.
Persistent web panel overlay in clinicIQ web app; mobile responsive V1, native V2
Decision 08
Monthly Master Plan Refresh
Layer 4 closes the loop back to Layer 2 monthly. How is this triggered? Fully automatic, or does it require expert initiation? What happens if the expert doesn't review it?
Automatic trigger on last day of month; expert reviews and approves before it takes effect

Development Philosophy

How We Build — Spec → Test → Eval → Ship While You Sleep

Every agent, every feature, every integration follows the same build loop. Engineers set it up, define what success looks like, and come back to results — not to babysit a process.

The Core Principle

Human time is for decisions, not execution. Engineers spec the task, define the test, write the eval function, and set it running. The system builds overnight. They come back to results, review what passed, fix what didn't, and ship. No one watches a progress bar.

"Set it. Sleep. Come back to results."
01
Phase 1 · Spec
Define the Task
Every agent starts with a written spec before a single line of code is written. The spec answers: what does this agent do, what are its inputs, what are its exact outputs, what does it read from, what does it write to, and what does failure look like.
Deliverable: Agent spec doc (already exists for all ~168 agents in Step 22)
02
Phase 2 · Test Suite
Build the Test Before the Agent
Before any agent is built, the engineer writes a test suite that defines what a passing output looks like. Input fixtures are prepared (sample dossiers, mock strategy docs, test briefs). Expected output structure and content rules are codified. The test runs against every build iteration automatically.
Deliverable: Input fixtures + output schema + test runner per agent
03
Phase 3 · Eval Function
Define What "Good" Means
The eval function is the automated judge. It runs every agent output through a defined scoring rubric — structural checks (did the output include all required fields?), quality checks (does the copy pass Voice DNA alignment?), and integration checks (does the output format exactly match what the downstream agent expects?). Outputs are scored and flagged — not just pass/fail, but graded so engineers see where quality breaks down.
Deliverable: Eval function per agent with weighted scoring rubric
04
Phase 4 · Overnight Build
Ship While They Sleep
Engineer commits the spec, test suite, and eval function. The build pipeline runs overnight — generating prompt variants, testing them against fixtures, scoring them through the eval function, iterating on failures, and surfacing the best-passing version for review. Engineer comes back to a ranked list of outputs with pass/fail scores, diff-highlighted against the spec, ready to approve or adjust.
Deliverable: Ranked output candidates with eval scores — human approves
What an Eval Function Looks Like
🏗 Structural Check
Did the output include all required fields? Is the JSON schema valid? Does every lesson script have all 6 required sections? Are all timestamps present in the podcast runsheet?
AUTOMATED — binary pass/fail
🎙 Voice Alignment Check
Does the copy match the Voice DNA sample extracts? Does it use first-person where required? Does the vocabulary complexity match the expert's baseline? Are ALWAYS/NEVER rules respected?
LLM-GRADED — scored 0–100
🔗 Integration Check
Does the output format exactly match what the downstream agent expects as input? Does the handoff payload include the required flags? Will the next agent in the chain be able to read this without error?
AUTOMATED — contract testing
📊 Quality Rubric Check
Does the content meet the spec's quality standard? For copy: does it have a hook? A mechanism reference? A CTA? For an ad: is the hierarchy correct? Is the hook in the first 3 seconds? For a protocol: is every item specific enough?
LLM-GRADED — scored 0–100
⚡ Regression Check
When a prompt or agent is updated, does the new version still pass every test the previous version passed? Regression suite runs automatically on every commit — no previously-working behavior gets broken silently.
AUTOMATED — runs on every commit
👤 Human Spot Check
For high-stakes outputs (sales scripts, webinar scripts, program protocols) — the final gate is a human review. Engineer or product owner reads the output against the spec. One approval unlocks production deployment.
HUMAN — required before prod deploy

How This Applies to clinicIQ Specifically
All 168 agents are pre-specced
Step 22 is the spec doc for every agent. Engineers don't start from a blank page — they start from a complete description of inputs, outputs, and success criteria. The spec IS the first deliverable. It's done.
Test fixtures come from real dossier data
The THI/Josh Axe expert profile is the gold standard test fixture. Every agent gets tested against real expert data — not synthetic toy examples. If it works on that profile, it works.
Layer 4 evals agents in production
The Learning & Feedback Engine (Step 20) is essentially a production eval function — it scores every agent output against performance data and human feedback continuously. The eval philosophy doesn't stop at deployment. It becomes the live quality system.
Agents build agents
For the prompt engineering and iteration work — the overnight build runner itself uses an LLM to generate prompt variants, interpret failure reasons, and produce targeted fixes. Engineers set the goal; AI does the iteration. This is how ~168 agents get built without ~168 person-months of work.
Human gates are explicit and minimal
The table above defines exactly which agents need human review before production deploy — and which ones the eval function gates completely. No ambiguity. Engineers know going in what they'll need to review and what will ship automatically.
Clinical outputs always have a human gate
Program protocols, supplement stacks, dosing recommendations — these always require a clinician or expert review before they reach a patient. No eval score overrides this. The eval gates quality; the human gates safety.
Pre-Launch Quality Gate

Miro Fish — Audience Simulation Before Every Launch

Before any ad, webinar, or funnel goes live, it runs through a synthetic crowd of the expert's specific avatar at each relevant level of awareness. It scores, it recommends, and it gets out of the way. If it passes — roll. If not — here's exactly what to fix. Then roll.

BUILDER AGENT produces finished asset ↓ OUTPUT QUALITY AGENT (Voice QA + Brand QA) ↓ MIRO FISH SIMULATION AGENT ◄── HERE ↓ ┌─────────────────────────────┐ Score ≥ 85 → 🟢 ROLL Score 70–84 → 🟡 FIX FIRST IQ Drive → Publish Auto-refine → Re-score → Publish (max 2 cycles, then expert decides)
🧑‍🤝‍🧑 The Synthetic Crowd
Built from the expert's Core Avatar (8-layer dossier profile), sampled across demographic variance. Crowd size is fully configurable — defaults by asset type, adjustable per run. IQ Claw shows estimated token cost before every simulation.

📊 What Gets Scored
5 universal dimensions: Hook Strength (25%), Avatar Mirror (20%), Mechanism Clarity (15%), Believability (20%), Action Intent (20%). Plus asset-type-specific dimensions — ad thumb-stop power, webinar false-belief dismantling, funnel above-fold clarity, etc.
🎯 The Verdict
85–100: Roll. Asset cleared — ship it.
70–84: Fix First. 1–2 specific issues, automated refinement, re-score.
50–69: Significant Rework. Multiple issues, 2 refinement cycles, then expert decides.
<50: Fundamental problem. Expert review required.
🔧 The Recommendations
Every failing dimension gets a specific, actionable fix — not "improve the hook" but the exact line, the exact crowd reaction it's triggering, and the exact rewrite. Specific enough that the Builder agent can execute without asking a clarifying question. Automated refinement cycle kicks off immediately.
⚡ Expert Override
One click. Always available. Override is logged, the asset ships, and Layer 4 monitors its real performance with heightened attention. If Miro Fish was wrong (overrode and it performed) — the model learns. If Miro Fish was right (overrode and it underperformed) — the model is reinforced. No decision is wasted.
📈 Gets Smarter Over Time
Layer 4 compares every Miro Fish prediction to real performance data post-launch. Accurate predictions reinforce the model. Wrong predictions update it. At platform level: anonymized patterns across experts in the same specialty improve default simulation parameters for every new expert in that niche.
